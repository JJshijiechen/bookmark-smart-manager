# 架构设计: Smart Bookmark Manager

## 技术栈

- Browser extension: Manifest V3
- UI: HTML, CSS, vanilla JavaScript
- Browser APIs:
  - `chrome.bookmarks`
  - `chrome.tabs`
  - `chrome.storage.local`

## 模块结构

```text
manifest.json     扩展配置、权限、入口页面和图标
popup.html        浏览器扩展弹窗入口
popup.js          弹窗搜索、过滤、保存当前标签、域名操作
dashboard.html    选项页/完整管理面板入口
dashboard.js      分组视图、批量选择、删除、重新分类、自动整理
utils.js          书签加载、分类规则、分组、重复检测、文件夹整理逻辑
styles.css        弹窗和 dashboard 的统一样式
icons/            扩展图标
```

## 数据流

1. UI 页面调用 `loadAllBookmarks()`。
2. `utils.js` 通过 `chrome.bookmarks.getTree()` 读取完整书签树。
3. 书签被展开为扁平列表，并补充 `domain` 和 `category` 字段。
4. 分类优先读取 `chrome.storage.local` 中的用户选择；没有手动分类时使用本地域名/关键词规则。
5. UI 根据搜索词、分类、域名或 dashboard 分组模式渲染列表。
6. 删除、保存、打开和移动操作直接调用浏览器书签或标签 API。

## 分类策略

分类逻辑是本地规则驱动：

- 先检查域名是否匹配分类规则。
- 再检查标题或 URL 是否包含分类关键词。
- 没有命中时归入 `Other`。
- 用户在 dashboard 中手动选择分类后，结果存入 `chrome.storage.local`。

## 自动整理策略

自动整理会把原始书签移动到书签栏下的顶层分类文件夹：

- 自动定位可见书签栏，优先使用 Chromium 常见 ID `1`。
- 为有内容的分类创建或复用文件夹。
- 对已经位于目标分类文件夹的书签跳过移动。
- 旧版本的 Smart Organized 文件夹会被清理或兼容处理。

由于自动整理会改变用户书签结构，UI 使用二次点击确认。

## 隐私与安全

- 不请求外部 API。
- 不上传书签标题、URL 或分类。
- 不使用远程模型或第三方服务。
- 删除和移动属于高影响操作，现有 UI 使用二次点击确认降低误操作风险。

## 可维护性

- 分类规则集中在 `utils.js` 的 `CATEGORY_RULES`。
- 共享书签处理逻辑集中在 `utils.js`，避免 popup 和 dashboard 重复实现。
- UI 层只负责状态、事件和渲染。
